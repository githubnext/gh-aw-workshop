# Step 9: Reading Workflow Output

> _An agentic workflow is only as useful as what you can learn from its output — knowing how to read the results is the skill that lets you improve fast._

## 🎯 What You'll Do

You'll learn how to interpret the three places where [agentic workflow](https://github.github.com/gh-aw/introduction/overview/) results appear: the live run log, the run summary, and the [safe-output](https://github.github.com/gh-aw/reference/safe-outputs/) record. By the end you'll know what a healthy run looks like and where to look first when something goes wrong.

## Where Does Output Live?

An agentic workflow produces output in three distinct places. Think of them as three different lenses on the same run.

| Location | What it shows | Best for |
|----------|--------------|----------|
| **Live log** | The agent's reasoning steps and tool calls, streamed in real time | Debugging |
| **Run summary** | A structured recap written by the agent after it finishes | Quick health check |
| **Safe-output record** | Every write action the agent actually performed (comments posted, issues created, etc.) | Audit and verification |

---

## Anatomy of the Live Log

Open the **Actions** tab, click your completed **Hello Agent** run, then click the job name to open the log.

The log entries follow a predictable pattern:

```
🤔 [plan]   Deciding what to do next
🔧 [tool]   github.list_issues  → {state: open, sort: reactions}
📥 [result] 3 issues returned
🤔 [plan]   Issue #4 has 7 👍 reactions — this is the target
🔧 [tool]   github.add_comment  → {issue_number: 4, body: "..."}
📤 [output] safe-output: add-comment  (1 of 1 allowed)
✅ [done]   Task complete
```

Each line type tells you something different:

- `[plan]` — the agent is reasoning. No API call has been made yet. If you see many `[plan]` lines followed by no `[tool]` lines, the agent may be confused by unclear instructions — ask the `agentic-workflows` skill to tighten the workflow body.
- `[tool]` — the agent is calling an external API. You can see exactly which API and what parameters were sent.
- `[result]` — the raw data that came back from the tool call.
- `[output]` — a write operation is about to be committed (or was blocked by a safe-output limit).
- `[done]` — the agent finished its task.

## Reading the Run Summary

Before you scroll to the **Summary** tab, make a quick prediction about what should appear in a healthy summary.

### Exercise: Predict before you read

- [ ] In one sentence, predict which three sections a healthy summary will contain.
- [ ] After writing your prediction, scroll to the run **Summary** section.

The summary is generated by the agent itself at the end of a successful run. A healthy summary looks like this:

```
### Summary

I found 3 open issues. Issue #4 had the most 👍 reactions (7).
I posted a comment on issue #4 summarising the top-voted requests.

**Safe outputs used:**
- add-comment: 1 / 1 allowed
```

Key things to check:

1. **Does the action described match what you asked for?** If you asked for a comment on the top-voted issue but the summary mentions a different issue, ask the `agentic-workflows` skill to revise the instructions in your workflow body.
2. **Safe outputs used** — this line shows you whether the agent hit its write limit. A `1 / 1` result means it used its one allowed write. A `0 / 1` means it either had nothing to write or was blocked.

> [!NOTE]
> <details>
> <summary>Not all runs produce a summary section. If the run fails mid-way, the summary may be empty or missing. In that case, the live log is your best diagnostic tool.</summary>
> </details>

## Verifying the Safe-Output Record

The safe-output record is the ground truth — what was actually written to GitHub, not just what the agent reported. To verify a run, open the issue the agent commented on and confirm the comment exists, the content matches your instructions, and the timestamp aligns with the run.

For a full walkthrough of how to audit safe-output records and trace mismatches back to their cause, see [Side Quest: Diagnosing Common Agent Output Patterns](side-quest-09-01-debug-output.md).

> [!WARNING]
> <details>
> <summary>Never edit a comment the agent created just to fix a typo. Instead, ask the `agentic-workflows` skill to fix the workflow instructions and re-run. Editing manually means the next run will post a second comment — now you have two.</summary>
> </details>

## If Output Looks Off, Check These First

- [ ] If the log has many `[plan]` lines and no `[tool]` calls, simplify the workflow instructions.
- [ ] If tool results are empty or denied, verify `permissions:` and your filters.
- [ ] If summary and GitHub state disagree, trust the safe-output record first.
